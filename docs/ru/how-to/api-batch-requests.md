# Batch requests

В этом руководстве показано, как позволить клиентам объединять несколько
логических операций (create/get/update/delete, а также actions
контроллеров) в один HTTP-вызов с помощью `restalchemy.api.batch`.

Каждый элемент батча повторно проигрывается как самостоятельный запрос
через *немодифицированный* стек routing/controller/packer — чтобы
поддержать батчинг, не нужно менять ничего в уже существующих подклассах
`Controller` или `Route`.

---

## 1. Подключение эндпоинта

Подключите `batch.BatchRoute` к корневому маршруту так же, как любой другой
маршрут:

```python
from restalchemy.api import batch
from restalchemy.api import routes


class Root(routes.RootRoute):
    v1 = routes.route(V1Route)
    batch = routes.route(batch.BatchRoute)
```

Это открывает единственный эндпоинт `POST /batch/`, который может
обращаться к любому ресурсу, достижимому из `Root` — отдельного
подключения на каждый ресурс не требуется.

---

## 2. Формат запроса

```json
{
  "requests": [
    {"method": "POST", "path": "/v1/vms/", "body": {"name": "vm-a"}},
    {"method": "GET",  "path": "/v1/vms/<uuid>"},
    {"method": "PUT",  "path": "/v1/vms/<uuid>", "body": {"name": "vm-b"}},
    {"method": "DELETE", "path": "/v1/vms/<uuid>"},
    {"method": "POST", "path": "/v1/vms/<uuid>/actions/poweron/invoke"}
  ]
}
```

Каждый элемент `requests`:

| поле      | обязательно | описание                                                    |
|-----------|-------------|--------------------------------------------------------------|
| `method`  | да          | `GET`, `POST`, `PUT` или `DELETE` — как в обычном запросе.   |
| `path`    | да          | Путь, который вы вызвали бы напрямую, включая query string (`?field=value`) для фильтрации. |
| `body`    | нет         | JSON-тело в том же формате, что ожидает реальный эндпоинт.    |
| `headers` | нет         | Дополнительные заголовки только для этого элемента.           |

---

## 3. Каждый элемент выполняется независимо, через весь стек middleware

Каждый элемент выполняется в best-effort режиме: ошибка одного не
останавливает и не откатывает остальные. Ответ всегда `200 OK` с парой
`status`/`body` на каждый элемент, даже если часть элементов завершилась с
ошибкой — проверяйте `status` каждого элемента отдельно, не полагайтесь на
общий HTTP-статус ответа.

Каждый элемент проходит через *весь* WSGI-стек middleware, оборачивающий
эндпоинт `/batch` (auth, per-route авторизацию/policy, rate limiting,
retry-on-deadlock и т.д.), а не только через внутренний слой
routing/controller. К элементу батча применяются ровно те же правила, что и
к прямому вызову того же пути: если middleware блокирует или ограничивает
по частоте прямой `POST /v1/vms/`, он точно так же заблокирует тот же
запрос, пришедший как элемент батча.

Следствие прохождения через весь стек на каждый элемент: если
`ContextMiddleware` приложения открывает транзакцию БД на каждый запрос,
каждый элемент получает *свою собственную* транзакцию — межэлементной
атомарности нет. Батч из пяти записей — это пять независимых транзакций,
как если бы вы сделали пять отдельных прямых вызовов; сбой в элементе 3
никак не влияет на элементы 1, 2, 4 или 5. Если вам нужно, чтобы несколько
записей успевали или падали вместе, это нужно реализовать как отдельную
операцию над целевым ресурсом (один эндпоинт, выполняющий все изменения в
одной транзакции), а не батчинг независимых вызовов.

---

## 4. Формат ответа

```json
[
  {"status": 201, "body": {"uuid": "...", "name": "vm-a", "...": "..."}},
  {"status": 200, "body": {"uuid": "...", "name": "vm-a", "...": "..."}},
  {"status": 200, "body": {"uuid": "...", "name": "vm-b", "...": "..."}},
  {"status": 204, "body": null},
  {"status": 200, "body": {"uuid": "...", "state": "on", "...": "..."}}
]
```

Ответ — JSON-массив, по одному элементу на каждый запрос, **в том же
порядке, в котором они были отправлены**. Каждый элемент повторяет то, что
вернул бы прямой вызов этого эндпоинта: реальный HTTP-код и разобранное
тело ответа (или `null` для пустого тела). Неуспешный элемент выглядит так
же, как обычное тело ошибки RestAlchemy:

```json
{"status": 404, "body": {"type": "RecordNotFound", "code": 404, "message": "..."}}
```

---

## 5. Ограничения и ошибки валидации

- `requests` должен быть JSON-массивом; иначе для всего вызова целиком
  бросается `ParseError` (HTTP 400).
- `BatchController.__max_batch_size__` (по умолчанию `100`) ограничивает
  число элементов в одном вызове; превышение вызывает
  `BatchSizeLimitExceeded` (HTTP 400) для всего вызова целиком, ещё до
  выполнения хотя бы одного элемента. Переопределите атрибут в подклассе,
  если нужен другой лимит.
- Элемент батча не может быть направлен на сам batch-эндпоинт
  (`NestedBatchNotAllowed`, HTTP 400) — такой вызов отклоняется вместо
  рекурсии.

---

## Резюме

- `batch.BatchRoute`/`batch.BatchController` переиспользуют существующее
  дерево маршрутов, контроллеры и packer-ы без изменений — батчинг
  полностью аддитивен.
- Каждый элемент — best-effort и независим: проверяйте `status` каждого
  элемента.
- Элементы проходят через весь стек middleware, поэтому auth/policy/rate
  limiting/retry ведут себя точно как при прямом вызове — ценой отсутствия
  межэлементной транзакции: у каждого элемента своя.
- Actions (`.../actions/<name>/invoke`) работают внутри батча точно так же,
  как при прямом вызове, поскольку элемент батча — это просто повтор
  path/method/body.
